# Excel Memory Agent (EMA)

**A safe Excel automation layer for AI agents.**

EMA is local, model-agnostic, MCP-compatible infrastructure that lets an AI agent make
**safe, verifiable edits to existing business workbooks** — the expense trackers, bookkeeping
ledgers, inventory sheets, and sales reports that real organizations already run on.

Every write is schema-validated, backed up, written atomically, verified after the fact, and
reversible with a single undo. Formula cells are protected and Excel Tables are extended in place.

> **Status:** Safe-write core complete and tested (100 passing tests, ≥90% coverage target on the
> writer). Service, MCP server, CLI, and the optional local-LLM front-end are in progress — see the
> [Roadmap](#roadmap). This README documents the system as designed in
> [`docs/architecture_final.md`](docs/architecture_final.md).

## The problem

AI agents are good at deciding *what* a spreadsheet should contain. They are dangerous at the
*mechanics* of changing one.

Hand an LLM a workbook and ask it to "add this month's expense," and the typical paths are all bad:

- **Regenerate the file.** The model rewrites the sheet from a CSV-like dump and silently destroys
  formulas, formatting, named tables, and any sheet it didn't think to mention.
- **Blind cell writes.** The agent guesses a row and column, writes into a formula cell or the wrong
  table, and you discover it three reconciliations later.
- **No safety net.** There's no backup, no verification that the write actually landed, and no way to
  undo. A bad edit to a tax or accounting workbook is a real incident, not a retryable API call.

The hard part of "AI + Excel" was never the reasoning. It's doing the write **safely** against a file
that already contains business logic people depend on.

EMA is that safety layer. The agent proposes a row; EMA validates, backs up, writes atomically,
verifies, and gives you an undo token. The reasoning engine never touches the file directly.

---

## Why existing workbooks are hard for AI agents

A greenfield data store is a clean slate. An *existing business workbook* is the opposite — it is a
live system with implicit contracts:

- **Formulas and derived columns.** A `Total = Qty * UnitPrice` column must never be overwritten with a
  literal. The structure must be respected, not flattened.
- **Excel Tables (ListObjects).** Appending isn't "write to the next empty row" — the table range has
  to be *extended* so banding, filters, and downstream references keep working.
- **Layout that isn't row 1.** Banner rows, merged cells, and headers starting on row 3 break naive
  "first row is the header" assumptions.
- **Pivots, charts, and macros.** These are fragile under programmatic rewrite and are routinely lost
  when a workbook is regenerated by a library that doesn't preserve them.
- **Concurrency and sync.** The file may be open in Excel or mid-sync in OneDrive/SharePoint. Writing
  into a locked or syncing file produces conflict copies and corruption.
- **No transactionality.** Excel has no rollback. If a save fails halfway, you can be left with a
  truncated, unopenable file.

A model that "just edits the file" has to get *all* of this right, every time, with no feedback loop.
That is not a prompt problem — it's an infrastructure problem.

## Why not just use SQLite?

It's a fair question, and the honest answer is: **for a greenfield application, SQLite (or Postgres)
is usually the better choice.** A real database gives you types, constraints, transactions, and
concurrency for free. If you're starting from nothing, don't put your system of record in a
spreadsheet.

EMA exists for the other, very common situation: **the organization already runs on Excel.**

- The monthly books, the inventory sheet, and the sales report are *already* `.xlsx` files with
  formulas, tables, and reporting logic built over years.
- Finance, ops, and management open these files in Excel directly — that workflow isn't going away.
- Migrating an entrenched workbook to a database is a project with cost, risk, and political weight.

In that world, "rewrite everything as a database" is not on the table this quarter. The realistic ask
is: *let an AI agent safely append to and update the workbook we already have, without breaking it.*
That's the niche EMA fills — **safe automation of existing Excel assets**, not a new storage engine.

## What EMA guarantees

These are the core invariants, enforced in code (primarily in `ema/excel_io.py`) and covered by tests:

| Guarantee | What it means |
|-----------|---------------|
| **Schema validation** | Columns, types, and positions are discovered once and re-validated against the *live* file before every write. Drift (a renamed/moved header) is detected and rejected, not written through. |
| **Backup** | Every write copies the workbook to a timestamped backup first. Backups are pruned to a configurable depth. The backup *is* the undo state. |
| **Atomic writes** | Writes go to a temp file, which is opened to confirm it's valid, then swapped into place via `os.replace`. A failed save never leaves a half-written workbook. |
| **Verification** | After saving, EMA reopens the file and confirms the new row actually contains the expected values. On mismatch it restores the backup and raises. |
| **Undo** | Each successful write returns an `undo_token`. One call restores the pre-write bytes. |
| **Formula protection** | Columns detected as formula-bearing are refused as write targets — the agent cannot clobber a derived column with a literal. |
| **Table awareness** | If the data lives in an Excel Table, the table range is extended by one row and the value is written inside the range, preserving table semantics. |

The reasoning engine (host LLM or local LLM) **proposes**; the service **decides**; the writer
**executes**. The LLM never imports or touches the file layer — the call graph forbids it.

## Scope and limitations

EMA is deliberately honest about its boundaries. Read this before adopting it.

- **EMA safely updates the *tabular regions* of business workbooks** — plain header+data ranges and
  Excel Tables.
- **EMA does *not* guarantee loss-free round-tripping of:**
  - pivots
  - charts
  - VBA / macros
- **Backups and undo are the safety mechanism.** EMA backs up before every write, so even if a feature
  it doesn't preserve is affected, the original file is always recoverable.

If your workbook's value is primarily in its pivots, charts, or macros, treat EMA's writes as
"append data to the tables, restore from backup if anything else shifted" — not as a perfect editor of
every Excel feature. See [Limitations](#limitations) for the full list.

## Architecture

EMA is a small Python package with one file per concern. Two interchangeable front-ends (an MCP host
and a local-LLM CLI) both produce the **same validated `AppendRequest`** and feed the **same safe
core**. There is never more than one reasoning engine in a single path.

```
                 (NL text)                       (structured args)
        ┌──────────────────────┐          ┌──────────────────────────┐
        │  Local LLM front-end │          │   MCP host front-end      │
        │  (CLI / standalone)  │          │  (Claude/Cursor → tools)  │
        └──────────┬───────────┘          └────────────┬─────────────┘
                   │   both emit a validated AppendRequest             │
                   └───────────────┬───────────────────────────────────┘
                                   ▼
                          ┌─────────────────┐
                          │   EmaService    │   orchestration + safety policy
                          └────────┬────────┘
                 ┌─────────────────┼─────────────────┐
                 ▼                 ▼                 ▼
            registry.py        schema.py         excel_io.py
            (JSON state)   (truth + contract)   (backup→atomic→verify→undo)
```

### Component overview

| Component | File | Responsibility |
|-----------|------|----------------|
| **Models** | `ema/models.py` | All Pydantic v2 contracts crossing module boundaries (`WorkbookSchema`, `AppendRequest`, `AppendResult`, extraction DTOs). |
| **Errors** | `ema/errors.py` | Flat, typed exception hierarchy. The library raises; front-ends translate. |
| **Config** | `ema/config.py` | Constants and env-overridable settings (`EMA_HOME`, backup depth, LLM toggles). |
| **Registry** | `ema/registry.py` | `RegistryStore`: atomic, human-readable JSON persistence of known workbooks and their schemas. No Excel knowledge. |
| **Schema** | `ema/schema.py` | Discover headers/tables/types, validate the live file against the persisted schema, and emit the extraction contract used by the LLM. The single source of truth for column shape. |
| **Writer** | `ema/excel_io.py` | The paranoid core: `check_writable → backup → append (table-aware) → atomic save → verify → undo`. Knows nothing about the LLM. |
| **Service** | `ema/service.py` | `EmaService`: the public API and primary test surface. Workbook resolution, default-sheet substitution, and the LLM safety gate. |
| **MCP server** | `ema/server.py` | Thin FastMCP wrappers over `EmaService`. Validates input, returns JSON, converts errors to structured dicts. |
| **LLM provider** | `ema/llm.py` | One `LLMProvider` Protocol, one real impl (`OllamaProvider`), one test `FakeProvider`. The only code that talks to a model. Optional dependency. |
| **Extractor** | `ema/extract.py` | Natural language → validated candidate row, with each field tagged `extracted` / `generated` / `default`. Never writes files. |
| **CLI** | `ema/cli.py` | The home of the local-LLM path: `ema add/register/list/undo`, preview-by-default with confirm. |

**Engineering constraint:** `extract.py` must stay smaller than `excel_io.py`, and the LLM code never
imports the writer. Safety is enforced by the dependency graph, not just by convention.

## Run modes

EMA is one core with three ways to drive it:

| Mode | Reasoning engine | Local LLM needed | Commit policy |
|------|------------------|------------------|---------------|
| **MCP (host)** — primary | The host's LLM (Claude Desktop, Cursor, …) | No | May commit directly; still backs up, verifies, and returns an undo token. |
| **Standalone CLI** | Local Ollama model | Yes (opt-in) | Preview by default; commits only on explicit confirmation. |
| **Library** | None — caller supplies a structured `AppendRequest` | No | Caller's responsibility. Fully deterministic and unit-testable. |

The core installs and runs with **no model and no cloud**. The LLM is an optional `[llm]` extra,
disabled by default.

## MCP workflow

EMA exposes a small set of MCP tools. A capable host (Claude Desktop, Cursor, etc.) does the reasoning
and calls these tools; EMA does the safe write.

| Tool | Purpose |
|------|---------|
| `register_workbook` | Register an `.xlsx` and propose/confirm its schema (`confirm=False` previews, `confirm=True` persists). |
| `list_workbooks` | Enumerate registered workbooks. |
| `get_workbook_schema` | Return the persisted schema for a workbook. |
| `append_row` | Safely append one row (`backup → atomic → verify → undo`). Supports `dry_run`. |
| `undo_last_append` | Roll back the most recent write. |
| `append_from_text` | *Gated, off by default.* EMA-side extraction for hosts that explicitly opt in. |

Typical sequence:

```
host: register_workbook(path="Bookkeeping.xlsx", confirm=False)   → schema proposal
host: register_workbook(path="Bookkeeping.xlsx", confirm=True)    → persisted
host: append_row(workbook_id="bookkeeping", values={...}, dry_run=true)  → preview
host: append_row(workbook_id="bookkeeping", values={...})         → written + undo_token
host: undo_last_append(workbook_id="bookkeeping")                 → restored (if needed)
```

The host's model never writes the file. It only calls tools; EMA enforces every guarantee.

## Local LLM workflow

For a fully local, no-host setup, the CLI pairs EMA with a local model via [Ollama](https://ollama.com).
Natural language is **extracted into a candidate row, previewed with provenance, and only written on
confirmation.**

```
ema add "Record office supplies expense 42.90 EUR at Staples on 2026-06-10" --workbook expenses
  → EmaService.append_from_text(confirm=False)
      1. load the workbook's schema → build the extraction contract
      2. local model returns JSON: Amount=42.90 (extracted), Vendor="Staples" (extracted),
         Date=2026-06-10 (extracted), Category="Office Supplies" (generated)
      3. safety gate: confirm=False ⇒ dry_run ⇒ NO file write
  → CLI prints the proposed row, marks generated fields, asks: commit? [y/N]
  → on confirm: write with backup → atomic save → verify → undo token
```

Every value the model *invented* (rather than read from your text) is labeled `generated` in the
preview and audited in the result, so fabrication is always visible before a byte is written.

## Example user flows

### Example 1 — Add an expense to `Bookkeeping.xlsx`

Suppose `Bookkeeping.xlsx` has an Excel Table with columns
`Date | Vendor | Category | Amount | Total YTD`, where `Total YTD` is a formula column.

**Via the CLI (local LLM):**

```bash
ema add "Paid 129.00 EUR to AWS for hosting on 2026-06-09" --workbook bookkeeping
```

Preview (nothing written yet):

```
Proposed row for Bookkeeping.xlsx [Ledger]:
  Date      2026-06-09     (extracted)
  Vendor    AWS            (extracted)
  Category  Hosting        (generated)
  Amount    129.00         (extracted)
  Total YTD —              (formula column — protected, not written)
Commit? [y/N]
```

On confirm, EMA backs up the file, extends the table by one row, writes `Date/Vendor/Category/Amount`,
**leaves `Total YTD` untouched** so its formula recalculates in Excel, verifies the row, and returns an
undo token.

**Via MCP (host LLM) or the library — the same structured request:**

```python
from ema.service import EmaService
from ema.models import AppendRequest

svc = EmaService()
result = svc.append_row(AppendRequest(
    workbook_id="bookkeeping",
    values={"Date": "2026-06-09", "Vendor": "AWS", "Category": "Hosting", "Amount": 129.00},
))
print(result.written_row, result.undo_token)
# If anything looks wrong:
svc.undo_last_append("bookkeeping")
```

If you'd tried to write into `Total YTD`, EMA raises a `ValidationError` and the file is left untouched.

### Example 2 — Add an inventory movement to `Inventory.xlsx`

`Inventory.xlsx` tracks stock movements with columns
`Date | SKU | Description | Change | OnHand`, where `OnHand` is a running formula.

```bash
ema add "Received 50 units of SKU-4471 widgets into the warehouse today" --workbook inventory
```

Preview:

```
Proposed row for Inventory.xlsx [Movements]:
  Date         2026-06-12   (generated: "today")
  SKU          SKU-4471     (extracted)
  Description  widgets      (extracted)
  Change       +50          (extracted)
  OnHand       —            (formula column — protected, not written)
Commit? [y/N]
```

The `today` date is clearly flagged as **generated**, not silently assumed. On confirm, the movement is
appended inside the table, `OnHand` keeps recalculating, and you get an undo token. A bad entry is one
`ema undo --workbook inventory` away from gone.

## Business examples

EMA is built for the workbooks organizations already maintain:

- **Bookkeeping / accounting** — append ledger entries to `Bookkeeping.xlsx` without disturbing YTD and
  balance formulas.
- **Expense tracking** — log expenses to `Expenses.xlsx` (plain range or Excel Table) with category
  inference previewed before commit.
- **Inventory management** — record stock movements in `Inventory.xlsx` while running-total formulas
  recalculate untouched.
- **Sales reporting** — append daily/period rows to a sales report whose summary tables and charts feed
  off the underlying data table.
- **Tax workbooks** — add line items to structured tax sheets with a backup taken before every write.

In each case the value proposition is the same: the agent contributes *data*; EMA preserves the
*structure and logic*.

## Installation

**Requirements:** Python 3.11+.

```bash
# Clone
git clone https://github.com/<your-username>/Excel-Memory-Agent-EMA.git
cd Excel-Memory-Agent-EMA

# Core install (no model, no cloud)
pip install -e .

# Optional: local-LLM front-end (adds Ollama support)
pip install -e ".[llm]"

# Optional: development + test tooling
pip install -e ".[dev]"
```

For the local LLM path, install [Ollama](https://ollama.com) and pull a small instruct model:

```bash
ollama pull qwen3:4b
```

**Runtime state location:** EMA stores its registry, backups, and log under `EMA_HOME`
(default `~/.ema`). Keep `EMA_HOME` **outside** any cloud-synced folder (OneDrive, Dropbox) to avoid
sync locks and conflict copies:

```bash
# example (PowerShell)
$env:EMA_HOME = "C:\ema-data"
# example (bash)
export EMA_HOME="$HOME/ema-data"
```

## Development

EMA follows a strict, core-first build order: the safe-write core is fully built and tested before any
LLM code exists. One file per concern; the LLM layer never imports the writer.

```bash
# Set up an editable dev environment
pip install -e ".[dev]"

# Run the full suite
python -m pytest

# Run with coverage on the safety-critical modules
python -m pytest --cov=ema.excel_io --cov=ema.service --cov-report=term-missing
```

Key conventions:

- **Contracts first.** All cross-module data is a Pydantic v2 model in `ema/models.py`.
- **The writer is precious.** Changes to `ema/excel_io.py` must keep backup/atomic/verify/undo intact
  and stay ≥90% covered.
- **Determinism.** No network or real LLM in the default test run; the LLM path is exercised through a
  `FakeProvider`. UTC timestamps are mocked where assertions depend on them.
- **Logging, not printing.** The MCP server logs to `EMA_HOME/ema.log`; it never writes to stdout
  (the MCP stdio transport uses stdout for protocol traffic).

## Testing

Tests run **offline and deterministically** — no MCP host, no real model — against **real generated
`.xlsx` fixtures** (never mocks of openpyxl).

```bash
# Fast default run
python -m pytest

# Include slower formatted-workbook flows
python -m pytest -m slow

# Real local-model smoke test (requires Ollama; excluded by default)
python -m pytest -m llm
```

**Current status:** `100 passed` — the foundations (models, registry, schema) and the safe-write core
(`excel_io`) are complete and green. Service, server, CLI, and LLM tests are scaffolded for upcoming
milestones (see [Roadmap](#roadmap)).

Safety properties are first-class test cases, not afterthoughts:

- `dry_run` leaves the file **byte-for-byte identical** (asserted via mtime + hash).
- A forced bad write is **fully recovered** via backup + undo.
- Formula-column and type-mismatch writes are **refused with the file untouched**.
- A simulated mid-save failure **preserves the original** file.
- Table appends land **inside** the extended table range.

**Markers:** `slow` (formatted/chart workbooks) and `llm` (real Ollama) are both excluded from the
default fast run. **Coverage gate:** ≥90% on `excel_io.py` and `service.py`.

## Roadmap

EMA is built in two release gates. The first is a complete, LLM-free, safe product on its own.

**Phase A — Foundations** ✅
Contracts (`models`, `errors`, `config`), atomic registry, and real `.xlsx` fixtures.

**Phase B — Safe-write core** ✅
Schema discovery/validation and the paranoid writer (`backup → atomic → verify → undo`, formula
protection, table awareness). ≥90% coverage target on the writer.

**Phase C — Structured product (Gate 1)** 🚧
`EmaService` (structured path) + the MCP server's five always-on tools. **First shippable build:**
safe, structured, MCP-driven, fully LLM-free. Target tag `v0.1.0`.

**Phase D — Local-LLM front-end** ⏳
`LLMProvider` + `OllamaProvider`, the extractor with provenance flags, the `append_from_text` service
method behind the confirmation gate, and the `ema` CLI.

**Phase E — Hardening & release (Gate 2)** ⏳
Optional gated MCP text tool, a real-model smoke test on a local Qwen3, and full docs. Target tag
`v0.2.0`: a complete safe, local, AI-powered automation layer for business workbooks.

## Limitations

In the spirit of not overselling, here is what EMA does **not** do:

- **Tabular regions only.** EMA writes to plain header+data ranges and Excel Tables. It is not a
  general-purpose Excel editor.
- **No guaranteed round-trip of pivots, charts, or VBA/macros.** Workbooks containing these may lose or
  alter those features on write. Treat backups + undo as the safety net for anything outside the data
  tables.
- **`.xlsx` only.** `.xlsm` (macro-enabled) is out of scope (`keep_vba=False`).
- **Append-oriented.** The first-class operation is safe row append. Arbitrary cell edits are not the
  current design target.
- **Single-process.** Concurrency is handled by refusing to write to a locked/open file
  (`check_writable`), not by a locking daemon. Don't point two writers at the same workbook.
- **Undo depth of one.** Undo restores the most recent write from its backup; it is not a full history
  with multi-step rollback (older backups are retained but undo targets the last write).
- **Type inference is a hint, not a gate.** Schema type detection samples a handful of cells; it informs
  validation but won't block a legitimate write on an ambiguous column.

If you're starting greenfield, prefer a real database (see
[Why not just use SQLite?](#why-not-just-use-sqlite)). EMA earns its place specifically when you must
work *with* an existing Excel workbook.
